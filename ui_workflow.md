# SmartProcure UI 開發工作日誌

本文件詳細記錄了基於 MapelAI 與 D&B 領域知識，並嚴格根據現有資料集 (`data.csv`) 欄位所進行的 UI 開發工作流程。

## 階段一：需求確認與資料盤點
* **目標**：提升採購決策品質，加強風險控管與合規性。
* **可用資料欄位盤點**：
  * `Supplier Risk` (Low/Medium/High)
  * `Supplier ESG Score`
  * `Maverick Spend` (Yes/No)
  * `On Time Delivery` (Yes/No)
  * `Days Late`

## 階段二：UI 實作 (已完成)
* **動態補水機制 (Data Enrichment)**：為解決前端使用者認知負擔，將表單輸入維度減少至 4 個，並由後端 API 自動查表補齊 6 個風險/評分欄位，達成模型預測。
* **總覽頁 (Overview)**：新增了「平均 ESG 評分」與「越權採購 (Maverick Spend)」的 KPI 卡片，讓決策者一眼看出合規與風險狀況。
* **供應商分析頁 (Supplier Analysis)**：重構了資料表欄位，將原先無法對應的欄位替換為資料集真實存在的 `Supplier Risk`、`Supplier ESG Score`、與交期 (`On Time Delivery` / `Days Late`)，並加入視覺化 Badge (Low/Medium/High)。
* **供應商推薦模型 (Recommendation)**：將原先的「合規優先」情境按鈕，調整為符合使用者需求的「低風險與合規導向 (Risk & Compliance)」。背後排序邏輯已正確綁定 `Supplier ESG Score`。
* **風險訂單清單 (Risk List)**：針對高風險或越權採購的訂單，新增了「轉向合規供應商」的快速處置按鈕，點擊後可一鍵跳轉並帶入合規推薦邏輯。

## 階段三：測試與交付
* 確認所有新增與修改的 DOM 元素符合既有 CSS 樣式與結構。
* 確認按鈕的 onClick 事件路由正常。
* 遵守了「無資料集對應欄位不新增」的開發鐵則。

## 階段四：新供應商准入評估 (Open Data 架構)
* **需求緣起**：針對缺乏歷史交易資料且無商用徵信(如鄧白氏)授權的新供應商，開發輕量級評估介面。
* **開發項目**：新增「供應商准入評估」視圖。
* **資料來源設計**：以 OpenCorporates 取代傳統商業統編查詢，並結合 AI 網路負面新聞檢索來動態生成評分。

## 階段五：全彩五角雷達圖與後端修復
* **需求緣起**：使用者希望以更具視覺衝擊力的方式呈現新供應商的各項能力維度。
* **圖表邏輯**：採用 `ApexCharts` 實作全彩五角雷達圖。當時因為 API 僅提供單一風險，五大維度採前端 Mock (隨機生成) 方式實作。

## 階段六：五角雷達圖全棧真實計分重構 (Authentic Radar Scoring Refactor)
* **需求緣起**：使用者察覺新供應商准入評估的雷達圖數值無法真實反映供應商狀態 (因階段五採用 Mock 隨機產生)。為建立決策者信任，必須將雷達圖的所有維度改為由後端真實計算。
* **無歷史資料的新供應商評估邏輯 (Data Source Annotation)**：
  * 對於全新供應商，系統**沒有內部歷史交易資料 (無真實 days_late 等特徵)**，因此所有分數皆為基於 **OSINT (開源情報)** 的 AI 推理與特徵工程：
  * **永續發展 (ESG)**：直接抓取 OSINT 負面新聞網搜，根據正面/負面關鍵字比例，計算出的精確 ESG 分數 (基礎分 70，負面詞扣分，正面詞加分)。
  * **合規性 (Compliance)**：基於 ESG 基礎分數。若被 OSINT 高度示警 (抓取到大量如 lawsuit, penalty, scam 等關鍵字) 觸發強制阻斷，直接重罰為 0 分。
  * **交期可靠 (Delivery)**：透過特徵工程估算 `days_late`。既然是新供應商，系統會分析 OSINT 新聞情緒。若負面爭議多，系統判斷其營運穩定度低，人為模擬給予較高的 `days_late` (例如 2~10 天延遲) 作為**風險代理指標 (Proxy Metric)**；再以此 `days_late` 反向扣除交期分數，完美銜接既有的機器學習特徵格式。這解釋了為什麼搜尋 Google 也會有交期延遲天數，因為那是 AI 根據新聞爭議量給出的營運延遲風險估算值。
  * **財務與定價**：由於無財報資料，以模型預測出的總體風險 (Risk Score) 以及地區、企業識別碼等屬性，給予不再浮動的穩定基準評估分數。
* **前端架構更新**：完全移除 `dashboard.js` 中的 `Math.random()`，雷達圖指標 100% 對接後端推論特徵。

## 階段七：真實財務數據串接架構 (Financial OSINT)
* **需求緣起**：使用者期望將雷達圖中的「財務 (Financial)」與「定價 (Pricing)」指標，從 AI 基準預測升級為串接真實的財務資料庫 (如資本額、是否上市櫃等)，並要求確保串接外部 API 不會導致前端讀取過慢或卡頓。
* **架構決策與設計**：
  * **外部資料源**：選用 `yfinance` 輕量化套件，並結合 Yahoo Finance Search API 進行公司名稱到股票代號的自動映射 (Ticker Mapping)。
  * **快取機制 (SQLite Caching)**：設計了 `finance_cache` 快取表。當成功查詢某公司財務資料後，會將結果寫入本地資料庫並設定過期時間。未來相同查詢皆可在 0.01 秒內完成，徹底解決 API 讀取延遲問題。
  * **嚴格超時與平滑降級 (Timeout & Graceful Degradation)**：為了極致化使用者體驗，財務查詢模組設定了嚴格的超時上限 (如 1.5 ~ 2.0 秒)。若遇到外部 API 當機或查詢未上市企業，系統將瞬間切斷連線並「平滑降級」，切換回階段六的 AI 風險代理分數，確保網頁永遠不會卡死，且最糟的等待時間也符合使用者對「AI 掃描」的情境預期。
  * **輕量擷取**：僅請求 `marketCap` (市值規模) 與 `quoteType` (上市櫃狀態) 兩個核心特徵，大幅降低封包傳輸與解析時間。


## 階段八：真實 ESG 權威數據串接與 OSINT 盲區修復
- **情境**：國際巨頭（如 Google、Intel）因其跨國規模，常在網路上產生大量新聞與訴訟。OSINT 若單純以負面關鍵字（如 lawsuit, penalty）計數，會導致大廠的 ESG 與合規分數遭到「誤殺」。
- **實作**：建立 esg_gatherer.py，作為真實國際評級（如 MSCI, Sustainalytics）的權威資料庫快取層。
- **防護豁免**：當主程式 (main.py) 判斷該供應商具備權威 ESG 評級時，自動套用真實高分（如 90 分以上），並 **主動解除 OSINT 負面字詞阻斷機制**，使得雷達圖能呈現「大到不能倒」企業真實的高穩定度。若為未上市企業查無權威 ESG，則無縫平滑降級，維持原有的 OSINT 嚴格網路探勘。

## 階段九：幽靈公司查核防呆與資料溯源透明度
- **需求緣起**：使用者發現輸入亂碼或查無實體的「幽靈公司」時，系統可能因為寬鬆的搜尋條件而誤發高分。此外，使用者希望能直接查閱 OSINT 新聞的原始來源以核實 AI 判斷。
- **雙層防呆防護網 (Fast-Fail Guardrails)**：
  1. **第一層 (Frontend)**：於 `dashboard.js` 加入純數字 (`999`) 與極短字串攔截，瞬間彈出警告，從根本避免無意義的 API 請求與後端浪費。
  2. **第二層 (Backend OSINT)**：重構了 `osint_gatherer.py` 內 Tavily 與 Wikipedia 的搜尋邏輯。除了嚴格化 boolean search (AND) 外，強制「搜尋結果內文必須完全包含輸入之公司名稱」，否則直接判定為「找不到該公司相關資訊」。若觸發此條件，`main.py` 會阻斷 ML 模型評分，強制作廢合規與財務分數，並回傳極高風險警告。
- **OSINT 溯源透視圖 (Source Transparency)**：
  - 修改 `main.py` 與 `SupplierRiskResponse` 模型，將 Tavily 回傳的原始新聞陣列 (`osint_sources`) 傳送至前端。
  - 於儀表板 UI 新增了「檢視原始新聞來源」的放大鏡 Icon 與彈出式 Modal (`dashboard.html` / `dashboard.js`)，強化分析結果的透明度與可解釋性。
